= Setup and Installation

include::common.adoc[]

== Requirements

To build the Q library, its tests, and its examples, you need:

. A C++20 compiler
. https://git-scm.com/[git]
. https://cmake.org/[CMake] 3.16.0 or higher
. A network connection the first time you configure (CMake downloads some
  dependencies; see <<_dependencies>>)

NOTE: The core of the library (`q_lib`), excluding the audio and MIDI I/O layer
(`q_io`), is header-only and depends only on a conforming C++20 standard
library. This core subset has been successfully used on bare-metal embedded
targets, including the Teensy and STM32 series, with no operating system.

=== Supported Platforms and Compilers

[cols="1,3", options="header"]
|===
| Platform | Toolchain

| macOS
| https://developer.apple.com/xcode/[Xcode] / Apple Clang (a recent version
  with full C++20 support)

| Windows
| Windows 10 or 11 with https://visualstudio.microsoft.com/vs/[Visual Studio 2022]

| Linux
| https://gcc.gnu.org/[GCC] 11 or later, or https://clang.llvm.org/[Clang] 11 or later
|===

The https://www.jetbrains.com/clion/[CLion] IDE works on all three platforms.

On Linux, the audio I/O layer (`q_io`) builds against ALSA, so install the ALSA
development headers first:

----
sudo apt-get install libasound2-dev
----

== Dependencies

Q depends on three libraries, brought in two different ways:

[cols="1,1,2", options="header"]
|===
| Dependency | Mechanism | Notes

| https://github.com/cycfi/infra/[Cycfi infra]
| git submodule
| Cycfi-owned support library. It ships as a submodule so you can edit it in
  place while developing (clone with `--recurse-submodules`). If the submodule
  is not checked out, CMake fetches it automatically, so a plain clone still
  builds.

| http://www.portaudio.com/[PortAudio]
| CMake FetchContent
| Cross-platform audio I/O. Downloaded automatically at configure time.

| http://portmedia.sourceforge.net/portmidi/[PortMidi]
| CMake FetchContent
| Cross-platform MIDI I/O. Downloaded automatically at configure time.
|===

You do not install any of these by hand. PortAudio and PortMidi are always
fetched by CMake; `infra` is used from the submodule when present and fetched
otherwise.

== Getting the Source

Clone the repository with its submodule:

----
git clone --recurse-submodules https://github.com/cycfi/Q.git
----

TIP: If you already cloned without `--recurse-submodules`, run
`git submodule update --init --recursive` to populate `infra`. (Even without
it the build works, falling back to fetching `infra`, but the submodule is the
recommended developer setup.)

== Building

From the repository root, configure and build with CMake:

----
cd Q
cmake -B build
cmake --build build
----

`cmake -B build` generates the build system into a `build/` directory using
your platform's default generator. The first configure downloads PortAudio,
PortMidi, and (if the submodule is absent) `infra`, so it takes longer than
subsequent runs. `cmake --build build` then compiles the library, the tests,
and the examples.

To build a single target instead of everything, name it:

----
cmake --build build --target example_sin_osc
----

=== Generating an IDE Project (optional)

CMake can emit a native project for your IDE instead of building directly.
Pass the matching generator with `-G`:

[cols="1,2", options="header"]
|===
| IDE | Command

| Xcode (macOS)            | `cmake -B build -G Xcode`
| Visual Studio (Windows)  | `cmake -B build -G "Visual Studio 17 2022"`
| CLion (all platforms)    | Open the top-level `CMakeLists.txt` directly
|===

Open the generated project in `build/` and build from the IDE as usual.

== Running the Tests and Examples

Run the test suite with CTest:

----
ctest --test-dir build
----

The examples are built by default. Each one lives in its own directory under
`example/` and is fully self-contained: examples that read audio bake the path
to their own `audio/` directory into the executable, so they run from any
working directory. Run one directly from the build tree, for example:

----
build/example/sin_osc/example_sin_osc
----

To check that everything is working, run `example_sin_osc`. With your system's
default audio output ready, it plays a five-second 440 Hz sine wave. You can
also run `example_list_devices`, which prints every audio and MIDI device it
detects. See the xref:tutorials/index.adoc[Tutorials] for a guided tour.

NOTE: You can turn off parts of the build with CMake options:
`-D Q_BUILD_EXAMPLES=OFF`, `-D Q_BUILD_TEST=OFF`, and `-D Q_BUILD_IO=OFF`
(the last drops the audio/MIDI I/O layer and its PortAudio/PortMidi
dependencies, leaving the header-only `q_lib` core).
