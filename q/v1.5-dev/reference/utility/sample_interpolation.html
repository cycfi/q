Sample Interpolation

Overview

The sample_interpolation namespace provides a family of interpolation policies for fractional (sub-sample) buffer indexing. Given a buffer and a fractional index, an interpolation policy computes a value between samples. The policies are used by the fractional_ring_buffer, and consequently by the fractional delays built on top of it, where delay times are not integer multiples of the sampling period, most importantly delay lines whose tap positions are modulated, as in chorus, flanger, vibrato, and pitch shifting effects.

The index convention follows the ring_buffer: index 0 refers to the newest element and increasing indices go back in time.

The figures below all interpolate the same set of samples, so the characters of the types can be compared directly.

The Interpolation Types

none

Figure 1. No Interpolation

No interpolation: the fractional part of the index is truncated and the sample at the integral index is returned, holding each value until the next, a zero-order hold. It requires no computation at all, but the staircase discontinuities translate to strong wideband error (~w/2 worst-case on a sinusoid with angular increment w). Use it when indices are known to be integral, or where speed trumps everything.

linear

Figure 2. Linear Interpolation

First-order interpolation between the two bracketing samples. It reproduces linear ramps exactly and its worst-case error on a sinusoid is ~w²/8, more than an order of magnitude better than truncation at low frequencies, at the cost of one multiply. It is the all-around default: the cheapest type with pass-through at the samples, suitable for most real-time applications.

cosine

Figure 3. Cosine Interpolation

A 2-point cosine ease with the same footprint as linear, but with a continuous first derivative, the slope is zero at every sample (note in the figure how the curve flattens momentarily at each point), and a guarantee none of the cubics can give: no overshoot. The output stays strictly within [y1, y2]. The ease, however, distorts in-band content (it is exact only at fractional positions 0, 0.5, and 1), so on band-limited audio it is less accurate than linear.

Its niche is therefore not audio but control-signal smoothing: automation, envelope breakpoints, morph positions, value noise: anywhere pass-through, C1 continuity, and bounded output matter more than spectral accuracy. The ease is computed as sin²(πμ/2) via the q sin lookup table (μ maps directly to the first quarter cycle, so there is no trigonometric call and no radian conversion), which makes it cheaper than the 4-point types.

cubic

Figure 4. Cubic Interpolation

4-point, third-order Lagrange interpolation. It passes through the samples and reproduces cubic polynomials exactly, making it the most accurate of the family on smooth signals. Its first derivative, however, is discontinuous at the samples, so it is best suited to taps that do not move; for modulated taps prefer hermite.

hermite

Figure 5. Hermite Interpolation

4-point cubic Hermite with Catmull-Rom tangents (central differences). It passes through the samples, reproduces quadratics exactly, and (unlike cubic) is C1-continuous across segment boundaries, so a tap sweeping through the buffer encounters no derivative discontinuities. This is the standard choice for modulated delay-line taps in chorus, flanger, vibrato, and pitch shifting effects.

bspline

Figure 6. B-spline Interpolation

4-point cubic B-spline. A smoother rather than an interpolator: it does not pass through the samples (at an integer index it yields (y[i-1] + 4·y[i] + y[i+1])/6), trading pass-through exactness for the best high-frequency rejection and smoothness (C2) of the cubics, clearly visible in the figure. It retains linear precision: constants and ramps are reproduced exactly. Use it when the buffer holds noisy data.

Accuracy comparison

Figure 7. Worst-case error reading a 1 kHz sinusoid (48 kHz sampling) at fractional offsets

The 4-point interpolators are the quality tier, two to three orders of magnitude more accurate than linear on band-limited content. cosine ranks below linear here (its virtue is smoothness, not accuracy), and bspline pays for its smoothing with in-band attenuation.

All types are stateless and branch-free, and all share one floor and one integer conversion for splitting the index. The 4-point types measure at roughly 2–3× linear, negligible on desktop-class CPUs, and still cheap in MCU inner loops, where the two extra memory reads typically matter as much as the arithmetic. cosine computes its ease through the q sin lookup table rather than std::cos (which would otherwise dominate its cost several times over); the table stores only the first quarter cycle (~1 KiB of read-only memory, recovered to a full cycle by branchless quarter-wave phase folding), small enough for tightly-coupled memory on MCUs. Note also that hermite measures as the fastest of the 4-point types (its coefficient structure optimizes particularly well), making the usual quality choice for modulated taps the cheap one, too.

The numbers come from docs/scripts/bench_interpolation.cpp; rerun it to recalibrate on other hardware.

Valid index range

The 4-point types read one sample on each side of the bracketing pair, so they require one extra sample of margin at both ends:

Types Valid fractional index range

linear, cosine

[0, size-2]

cubic, hermite, bspline

[1, size-3]
Indexing outside the valid range is undefined behavior: the index wraps around the ring buffer and yields unrelated samples.

Expressions

Notation

X

Any sample_interpolation type, e.g. sample_interpolation::hermite.

x

Object of type X.

buf

A buffer indexable with std::size_t, where index 0 refers to the newest element, e.g. ring_buffer<float>.

i

Fractional index. Object of an arithmetic type I, e.g. float.

Function Call

Expression Semantics Return Type

x(buf, i)

Compute the interpolated value at fractional index i. none truncates the fractional part; all other types interpolate between the bracketing samples (see the Overview table for each type’s properties).

I
All policies are stateless: interpolation is a pure function of the buffer contents and the index. The fractional_ring_buffer invokes the policy from its index operator and converts the result to its value_type.

Example

// A fractional delay whose moving taps read through hermite
// (Catmull-Rom) interpolation:
using shifter_delay = basic_delay<
   fractional_ring_buffer<
      float                            // value type
    , std::vector<float>               // storage
    , float                            // index type
    , sample_interpolation::hermite    // interpolation
   >>;
